12.6 FILE OUTPUT

OPENING AND WRITE TO A FILE

ofstream BASICS
- ofstream (output file stream) is a class for writing to files. It inherits from ostream, the same base class cout belongs to, whiich is why you cant write to a file using << the same way you write to a console

THE FOUR STEP BASICS
1. Declare an oftream object: ofstream outFS;
2. Open the file by passing a string filename: outFS.open("HelloWorld.txt")
3. Check is_open() before writing anything
4. Write with <<, then call close() when done

WHY CHECK is_open()'
- if the file fails to open (bad permissions, full disk, etc.) and you skip the check, the program oes not crash or throw an error, it just silently fails. 
- It just silently fails.
- The stream enters a failed state and any << writes after that becomes no-ops, meaning nothing gets written and you get no warning
- This is why you check is_open() and return early (or otherwise bail out) if it's false, instead of letting execution continue into the wirte code.

TIMING OF WRITES
- data can only be written between a successful open() and a close().
- Not before opening and not after closing

QUICK ANALOGY
- ofstream is like sealing a letter in an envolope
- You cant put anything in before you open it, and once youve closed it back up you're done writing

EXAMPLE: WRITING A TEXT FILE

WRITING A TEXT FILE TAKEAWAYS
- When open() is called and succeeds, the file is created immediately on disk, even before any data is written.
- It starts out empty.
- This means writing data and creating the file are two separate things, creating happens at open(), not at the first <<

CONTROL FLOW MATTERS
- If is_open() returns false and the code returns early (like return 1; inside the if block), nothing after that point executes, including close().
- This is exactly why the early return pattern matters
	- It skeps the write and close logic safely instead of trying to open on a stram that never opens

ALWAYS CLOSE
- An ofstram should always be closed with close() once you're done writing.
- This flushes any buffered data to disk and rleases the file so other programs (or parts of your own program) can access it

EXAMPLE: WRITING A SIMPLE HTML FILE

WHY ostream& MAATERS: POLYMORPHISM THROUGH INHERITANCE
- writeHTML takes a parameter of type ostream&, not ofstream&.
- This is the key design choice in this example
- Since ofstream inherits from ostream, and ofstream object IS an ostream underneath, just with extra file-specific function added (open(), close(), is_open()).
- And cout is itself just a pre-built ostream object that the standard library wires up to point at the consel
- Becuase the functions parameter is the mroe general base type (ostream&), it can accept either an ofstream object (like outFS) or cout, since both satisfy "is an ostream"
- If the function had instead been written with the parameter type ofstream&, only actual ofstream object could be passed in, and writeHTMLFile(cout, htmlParagraph); would fail to compile

TAKEAWAY PATTERN
- Write functions against the base class type when you want the function to be reusable across multiple derivbed types

WHY writeHTMLFile SHOULD NOT CALL close() INTERNALLY
- two reasons stack here
	1. DESIGN RESPONSIBILITY:
		- The functions job is to write content, not manage the stream's lifecycle.
		- The caller (main) should decide when to close, since it might need to do mroe with the stream first
	2. TECHNICAL REASON
		- close() is only defined on ofstream, not the base ostream class (closing only make sense for files, not console output.
		- Since the parameter tpye is ostream&, calling .close() inside the function would fail to compile, because that method doesn't exist on the base type

THE << OPERATOR
- the << operator is what allows writing data (strings, numbers, etc.) into any ostream- derived stream, whether that's a file or the console
