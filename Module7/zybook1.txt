12.2 INPUT STRING STREAM

INPUT STRING STREAM
- Sometiems a programmer wishes to read input data from a string rather than from a keyboard. 
- A new input string stream variable of type istringstream can be created that reads input from an associated string instead of the keyboard (standard input).
- istringstream is derived from istream.
- An istringstream can be used just like the cin stream
- istringstream lets you treat a string exactly like a keyboard input, token by token, without you having to slice it apart

USING GETLINE() WITH STRING NAMES
- A common use of string streams is to process user input line-by-line
- The program below reads in a line as a string and then extracts individual data items from the string.
- The getline() function reads an input line into a string, and inSS.str(linestring); uses the str() fuction to initialize the stream's buffer to string lineString.
- Afterwards, the program extracts input from INSS using >>, The statement inSS.clear();
- The .clear() necessary to reset the state of the stream so that subsequent extractions start from the beginning of the input strings.
- In Detail
	- str() changes the data,
	- .clear() reset the flags

REACHING THE END OF A STRING NAME
- a programmer will not always know how much data exists in a user input string, so using multiple individual extractions (Ex: inSS >> data; inSS >> data2;) is not useful for reading all of the input data
- Input streams have a Boolean function called eof() or end of file that returns true or false depending on whether or note the end of the stream has been reached. 
- An if statement or while loop can check if the end of input string stream has been reached by using the extraction operator.
- In detail
	- eof() and the boolean value of inSS >> data are answering two opposite phrasing of bascially the same situation, that's where the confusion is coming from
	- eof() returning "true" mean "yes, I have reached the end." That's a statement about the stream's status
	- inSS >> data evaluating to true or false in the while condition means something is different: it's answering "should the loop keep going?" and the answer to that is "no" once we've hit the end, because there's nothing left to read.
		- so it resolves to false.
	- In a sense, while there is still data left to extract, eof() is false and inSS >> data is true (read successed, keep looping).
		- Once you run out of data and try one more extraction, eof() flips to true AND inSS >> data flips to false
			- They flip in opposite directions but at exact same moment.	
- An alternative way is to use the good() boolean
- There is a small chance that the extraction operator will trigger a read failure.
- The good() function returns false whether there is read failure or an end-of-file condtion. The alternate way requires some extra lines of code.

NOTES:
- WHAT AN ISTRINGSTRAM IS
	- istringstream lets you treat a string like keyboard input. Instead of cin pulling from the keyboard, you pull from a string sitting in memory using the same >> extraction syntax.
	- This beasts manually parsing with find/substr because >> automatically skips whitespace and grabs the next token regardless of length, no hardcoded index position needed.
	- declare and load at once:
		
		istringstream inSS(mySTRLine);
	- or declare empty and load later:
		
		istringstream inSS;
		inSS.str(lineString);

- WHY DECLARE IT ONCE OUTSIDE A LOOP INSTEAD OF FRESH EVERY ITERATION
	- both work, but constructing a new istringstream object every loop iteration means setting up and tearing down that objects repeadly, wasted overhead. Declaring it once outside the loop and refulling it each iteration is more efficient

- str() VS clear()
	- thiese do two completely different jobs:
		- .str(lineString) swaps the actaully data in the stream's buffer
		- .clear() resets the streams internal state flags (like oef/fail), it does not touch data
	- if you call .str() without .clear() in a loop the stream still thinks it's at eof aor in a fail state from the previous iteration, even though fresh data was just loaded in. That's why both calls are neeeded togethor
		
		inSS.clear();
		inSS.str(lineString);

- THE WHILE  (inSS >> data) PATTERN

	while (inSS >> data) {
		cout << data << endl;
	}

	- INss >> data doesnt return a literal true/false, it returns the stream itself, which gets evaluated in a boolean context based on its internal flags. While theres data left, extraction succeeds and the expression acts as true. 
	- Once you try to extract past the end, the extraction fails, eof flag flips to true, and the expression acts as false, ending the loop
	- Key point:
		- eof() == true and (inSS >> data) == false describe the same moment, just phrased oppositely. One say "yes I'm done" the other says "no, don't continue"

- good() VS THE SHORTCUT
	
	inSS >> data;
	while (inSS.good()) {
		cout << data << endl;
		inSS >> data;
	}
	cout << data << endl;

	- this version reads one value ahead of printing it
	- The loop body always prints the previous successful read, nver the newest one.
	- When the file extraction fails (nothing left),
	- Data keeps its last successfully reads value rather than getting overitten with garbage.
	- Thats why cout after the loop closes , to catch that final leftover value that the loops body never got to print

12.3 OUTPUT STRING STREAM

OUTPUT STRING STREAM
- an output string stream variable of type ostringstream can insert characters into a string buffer instead of the screen (standard output).
- ostringstream is derived from ostream.
- A program can insert characters into an ostringstream buffer using <<, just like the cout stream.
- The ostringstream meber funtion str() return the contents of an ostringstream buffer as a string

OSSTREAMBASICS
- ostringstream mirros cin/cout but writes to a string buffer instead of the screen, using << the same way. Pull the result out with .str()/

	osstringstream outSS;
	outSS << carSpeed;
	myStr = outSS.str();

- the value: a function that builds output into a stream and returns .str() gives the caller a reusable string, instead of cout output that just appears and disappears

FIXED + SETPRECISION
- setprecision(2) alone means "2 significant digits total" paring it with fixed changes that it to "always exactly 2 digits after the decimal" giving consistent output like 105.12.
- setprecision(0) with fixed rounds to whole numbers. Bot hare sticky, then stay in effect until changed

SETW
- sets a minimum width for the next single value only padding with spaces (right-aligned by default). Not sticky, must be repeated before each value

SETFILL
- changes the padding characters setw uses (default space). 
-setfill("-") then setw(15) << "" produces 15 dashes, a quick way to draw a divider line

12.4 FILE INPUT

OPENING AND READING FROM A FILE
- sometimes a program should get input from a file rather than from a user type on a keyboard,
- To read file input, a programmer can create a new input stream that comes from a file, rahter than the predefined input stream cin that comes from the standard input (keyboard).
- an input stram can then be used just like cin

READING UNTIL THE END OF THE FILE
- A program can ready varing amounts of data in a file by using a loop that reads until the endf of the file has been reached.
- The eof() function returns true if the previous stram operation reached the end of the file
- Errors may be encountered while attempting to read from a file, like the inability to read a file, reading corrupt data, etc.
- So a program should chck that each read was successful before using the variable to which the read data wass assigned. The fail() function retuns true if the previous stream operation had an error.

INPUT STREAM ERRORS
- A stream error occurs when insertion or extraction fails, causing the stream to enter an error state,
	- Example:
		- If a file as the string two but the program attempts to extract an intteger, the extraction will fail and the stream will enter an error state.
- An input stream may also enter an error state if a value extreacted is too large or small to fit the given variable. 
- While in an error state, an input stream may: skp extraction, set the given variable to 0, or set the given variable to the maximum (or minimum) value of taht variable's data type
- A stream internally uses sveral 1-bit error flags to tract the state of the stream
- A program can check a stream's error state using several stream function that return the current state. A stream error state is cleared using clear. 
- goodbit
	- indications no error flags are set and the stream is good
		- good() returns true if no stream erros have occured
- eofbit
	- indicates if end of file reaches on extraction
		- eof() returns value of eofbit, if end of file reach on extraction
- failbit
	- indicates a logical error for the previous extraction or insertion operation.
		fail() returns true if either failbit or badbit is set, indicating an error for the previous stream operation
- badbit
	- indicates an error occurred while readcing or writing the stream, and the stream is bad, Further operation on the stream will fail
		- bad() returns true if badbit is set, indicating the stream is bad. 

- Key gotcha from the walkthrough
	- goodbit is not its own independent flag. It just represents "no other flags are currently set." If eofbit, failbit, or badbit is on, goodbit is automatically off.
	- eofbit means the stream actually ran out of data. failbit means the last operation had a logical error (wrong type, bad parse, etc), even if there's still more data left in the file. These are not the same thing, and mixing them up leads to bugs like infinite loops if you only check eof() when bad data (not end of file) is what's actually breaking the read.
	- eof() is meaningless on output streams like cout, since output streams don't run out of data to read, there's no source to run dry.
