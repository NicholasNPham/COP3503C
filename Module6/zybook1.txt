
3.15 DEBUGGING
- DEBUGGING: is the process of determining and fixing the cause of a problem in a computer program
- TROUBLESHOOTING: is another word for debugging.
- Far from being an occasional nuisance, dubugging is a core programmers task, like diagnosing is a core medical doctor task
- Skill in carrying out a methodical debugging process can improve a programmer's productivity
- A common error amoung new programmers is to try to debug without a methodical process, instead of staring at the program, or making random changes to see if the output is improved.

- QnA Notes
	- The first step in debugging is to come up with a methodical process with prediciting the problem
	- A common predicted-problem testing approach is to insert print statements
	- The language is case sensitive, so using uppercase referes to a different variable
	- The list of all possible probelms is typically huge. A programmer thinks of a few possible problems, then test the most likely first. Testing may lead to new predictions
	- In contrast, beginning programmers commonly stare at the code or make random changes. A good skill to develop is to think methodically. Predict likely problems, then carefully test to validate each prediction

- THE METHODICAL PROCESS
	- Predict a possible cause, conduct a test to validate it, repeat
	- A common beginner mistake is staring at code or making random hoping something improves
	
- COMMON DEBUGGING TECHNIQUES
	- Insert print statements to observe variable values
	- Manually set a variable to a known value to isolate a section
	- Comment out unused code temporarily
	- Mark temporary debug statements with //FIXME DELETE so you dont forget to remove them 

- KEY TAKEAWAYS 
	- Dont list every possible problem first: pick the ost likely ones and test the first. Testing often leads to new predictions
	- Integer overflow is a silent bug
		- The program compiles and runs but produces wrong output when a value exceeds the data type's max capacity. Fix it by choosing a data type with enough range like "long long"

10.8 ADDITIONAL MATERIAL: A FIRST LINKED LIST 
- KEY ADVANTAGES OVER ARRAYS/VECTORS
	- A linked list allows items to be inserted anywhere in the list without shifting subsequent elements unlike arrays or vectors

- NODE STRUCTURE
	- Each node holds two things: the actual data value, and a pointer to the next node in the list

- HEAD NODE
	- A special node that marks the front of the list. It doesnt store real data (holds - 1 as a sentinel). The last node's nextNodePtr is set to nullptr to signify the end

- DEFINITION
	- A linked list is a linear data structure that contains nodes. Each node holds a data vlaue and a pointer to the next node. The head node marks the start, and the last node's pointer is set to nullptr to make the end:

- InsertAfter (3 STEPS)
	- Save the original next pointer (C) into tempNext
	- Point current node (A) to the new node (B)
	- Point new node (B) to C using tempNext

- KEY TAKEAWAY
	- When inserting a node, no elements move in memory. Only pointer values are update. This makes linked list more efficent for insertion than arrays or vectors. 

- MANAGING LARGE NUMBERS OF NODES:
	- Instead of declaring a named variable for every node (which is unscalable), you cna use poitner variables 
		- CurrObj
		- lastObj
	
	- lastObj acts as a bookmark that always tracks the last node in the list (the tails) so you never have to traverse the whole list to find where to append 

	- THE LOOP DOES THREE THINGS EVERY ITERATION

		// Assume currentObj, head, and lastObj is init already

		for (i = 0; i < 20; ++i) {        
		  currObj = new IntNode(rand()); // 1. Create a new node
      
		  lastObj->InsertAfter(currObj); // Append to the end
		  lastObj = currObj;             // Update the tail
		}

	- This scales to any number of nodes without adding new variables

	#include <iostream>
	#include <cstdlib>
	using namespace std;

	class IntNode {
	public:
	   IntNode(int dataInit = 0, IntNode* nextLoc = nullptr);
	   void InsertAfter(IntNode* nodeLoc);
	   IntNode* GetNext();
	   void PrintNodeData();
	   int GetDataVal();
	private:
	   int dataVal;
	   IntNode* nextNodePtr;
	};

	// Constructor
	IntNode::IntNode(int dataInit, IntNode* nextLoc) {
	   this->dataVal = dataInit;
	   this->nextNodePtr = nextLoc;
	}

	/* Insert node after this node.
	 * Before: this -- next
	 * After:  this -- node -- next
	 */
	void IntNode::InsertAfter(IntNode* nodeLoc) {
	   IntNode* tmpNext = nullptr;
   
	   tmpNext = this->nextNodePtr;    // Remember next
	   this->nextNodePtr = nodeLoc;    // this -- node -- ?
	   nodeLoc->nextNodePtr = tmpNext; // this -- node -- next
	}

	// Print dataVal
	void IntNode::PrintNodeData() {
	   cout << this->dataVal << endl;
	}

	// Grab location pointed by nextNodePtr
	IntNode* IntNode::GetNext() {
	   return this->nextNodePtr;
	}

	int IntNode::GetDataVal() {
	   return this->dataVal;
	}

14.4 ADDING OUTPUT STATEMENTS FOR DEBUGGING	

- Recursive functions can be particularly challenging to debug. Adding output statements can be helpful
- Furthermore, an additional trick is to indent the print statments to show the current depth ofrecusion.

- NOTES
	- debugging recursive functions is hard because the same function runs multiple times on the call stack at different depths so a wall of identical print statments gives you no idea which call produced which output
	- The trick is to track recursion depth visually by adding an indentAmt paremeter to the function. Every print statement prepends indentAmt so you can see which depth you are at.
	- Each recursive call concatenates more spaces ontin indentAmt (like indentAmt + "   "), so each level deeper gets more indenting in the output
	- The indentation is basically a visual call stack
		- Deeper indent means deeper in the recusion, and as the function returns back up the indentation shrinks, shwoing the stack unwinding.
	- Some programmers leave the debug print statement in the code and just comment them out with // when not in use, since they also serve a documentation

ZyDE 14.4.1: OUTPUT STATEMENTS IN A RECURSIVE FUNCTION
	
		#include <iostream>
		#include <string>
		#include <vector>

		using namespace std;

		/* Finds index of string in vector of strings, else -1.
		   Searches only with index range low to high
		   Note: Upper/lower case characters matter
		*/

		int FindMatch(vector<string> stringsList, string itemMatch,
					  int lowVal, int highVal, string indentAmt) { // indentAmt used for print debug
   
		   int midVal;        // Midpoint of low and high values
		   int itemPos;       // Position where item found, -1 if not found
		   int rangeSize;     // Remaining range of values to search for match
   
		   cout << indentAmt << "Find() range " << lowVal << " " << highVal << endl;
   
		   rangeSize = (highVal - lowVal) + 1;
		   midVal = (highVal + lowVal) / 2;
   
		   if (itemMatch == stringsList.at(midVal)) {   // Base case 1: item found at midVal position
			  cout << indentAmt << "Found person." << endl;
			  itemPos = midVal;
		   }
		   else if (rangeSize == 1) {                   // Base case 2: match not found
			  cout << indentAmt << "Person not found." << endl;
			  itemPos = 0;
		   }
		   else {                                       // Recursive case: Search lower or upper half
			  if (itemMatch < stringsList.at(midVal)) { // Search lower half, recursive call
				 cout << indentAmt << "Searching lower half." << endl;
				 itemPos = FindMatch(stringsList, itemMatch, lowVal, midVal, indentAmt + "   ");
			  }
			  else {                                    // Search upper half, recursive call
				 cout << indentAmt << "Searching upper half." << endl;
				 itemPos = FindMatch(stringsList, itemMatch, midVal + 1, highVal, indentAmt + "   ");
			  }
		   }

		   cout << indentAmt << "Returning pos = " << itemPos << "." << endl;
		   return itemPos;
		}

		int main() {
		   vector<string> attendeesList(0); // List of attendees
		   string attendeeName;             // Name of attendee to match
		   int matchPos;                    // Matched position in attendee list
   
		   // Omitting part of program that adds attendees
		   // Instead, we insert some sample attendees in sorted order
		   attendeesList.push_back("Adams, Mary");
		   attendeesList.push_back("Carver, Michael");
		   attendeesList.push_back("Domer, Hugo");
		   attendeesList.push_back("Fredericks, Carlos");
		   attendeesList.push_back("Li, Jie");
   
		   // Find "Aaron, Joe"
		   attendeeName = "Aaron, Joe";
   
		   // Call function to match name, output results
		   matchPos = FindMatch(attendeesList, attendeeName, 0,
								attendeesList.size() - 1, "   ");
		   if (matchPos >= 0) {
			  cout << "Found at position " << matchPos << "." << endl;
		   }
		   else {
			  cout << "Not found. " << endl;
		   }
   
		   return 0;
		}

- 14.4 ADDING OUTPUT STATEMENTS FOR DEBUGGING
	- running the correct code, searching for "Aaron, Joe" keeps going lower half until it hits a range of 0 0 and correctly returns -1 (not found)
	- -1 is a sentinal value, meaning a special agreed-upon value that signals "not found" rather than a real position.
	- When you change itemPos = -1 to itemPos = 0 the function no longer has a way to communicate failure. Instead it returns 0, which is a real position, so the caller thinks the search succeeded.
	- The bug isnt in the search logic itself, its's in how the result is communicated back up the call stack
	- The indented debug output made the bug traceable. Without it you'd you see "found at position 0" with no idea where in the recursion it went wrong. With it you can pinpoint exactly which depth and which case produced the bad value.
