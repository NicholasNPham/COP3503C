LINKED LIST
- a non-continguous data structure that stores nodes 
	- Node is a generic term for some unit of data in a data structure.
- Unlike arrays, there is no random access to any element.
- Data is accessed linearly, one node at a time, typically starting from the first node in the list.
	- Want to access the 3rd node? Start from the first, then go to the second, then the 3rd.
- This is often described as iterating through the list or traversing the list.

PEOPLE IN A LINE
- what would this look like in an array?
	- Pretty Standard Stuff
- People in a line: Linked List Edition
	- In a linked list, the data will likely be fragmented, and non-continguous
	- How can we access data if we dont know where it is?
	- Solution: Store the pointers to those location
	- Each person in the line will know only who follows them
	- We can store that connection as a pointer.
	
NODES
- A node is just a generic piece of some complex structure.
	- It will be implemented as a class/struct
- A node will generally contain:
	- Some amount of data (the stuff you  really care about)/
	- A connection (pointer) to one or more other nodes.
- Example Syntax:

	struct Node
	{
		SomeDataType theData; // What are we storing? ------------------ The actual data inside a node is ultimately the most important part - Nodes, and pointers to nodes, are a means to an end.
		Node* next; // What node comes after this one? ----------------- All linked list operations involved using Node pointers to create, delete, or acess existing Nodes
																	L_______ Nodes will be dynamically allocated and deallocated with new and delete.
	};

CREATING A LINKED LIST
- The first node:
	- Linked lists start by accessing some first node, often called the head.
	- To access this (or any other node), we use a node poitner
	- Example Syntax:

		Node* head = nullptr; // The start of everything!
		head = new Node; // Create the first node in the list

	- Since it's a pointer, it doesnt actually have any data itself... we have to allocate a Node object, with new
	- For every piece of data:
		1. A new Node must be allocated
		2. It must be connected to one more more other Nodes (if there are any others)

GETTING THE END OF THE LIST
- Only tracking the head node is not idea - adding new nodes requires traversing, or "traveling" to the end
- How do we write code that can handle any of these scenarios
- We have two options:
	1. Traverse over all nodes until we reach the last node - The one with a next pointer that points to nullptr (i.e., no one follows them in line)
	2. Store and update a pointer to the last node, a tail pointer. (generally the better choice!)

TRAVERSING THE LIST
- Many linked list oeprations will involve moving through some (or all) of the nodes, one at a time.
- No matter what we want to do with the data, we have to first get to it.
- The general algorithm for that is
	1. Start by creating a pointer to the first node // Remember: you want to create a pointer to something that already exists. You do not want to allocate a "new" Node or craete a copy of something
	2. If were pointing to a valid node... (or some other condition)
	3. Use the node and/or move to the next node (pointer = pointer->next).
	4. Go to step 2.

SINGLY-LINKED LIST OR DOUBLE-LINKED LIST?
- Two implementating: singly linked, or double linked.
- A singly-linked list is unidirectional access
	- Node have a single connectioppn, a next pointer
	- A pointer to the head (maybe a tail pointer)
	- Start from the beginnning, traverse in one direction
- A doubly-linked list is bidirectional.
	- Node have a double connection, next and previous pointers.
	- A head pointer ,and a tail pointer, and previous pointers.
	- A head pointer, and a tail pointer (to the start at the end)
	- Start at either end, traverse in either direction
- Optional (for either): The list could be circular
	- The last node's next points to the first (instead of nullptr)
	- Doubly linked list: First node's previous points to the last.

TRADE OFFS
- LINKED LISTS VS ARRAYS
	- Linked Lists
		- Adding or removing a node requires changing a small number of pointers "near" that node
		- Iterating over nodes one at a time to "walk to" a specific node can be slow, especially for large lists
		- Nodes store data and pointers to nodes
			- Though small, extra pointers means extra memeory (may be negligible)

	- Arrays
		- Requires creating an entirely new array for the new size and copying all elements
		- Random access of an array let's you jump to any elements - doesnt get much faster than this!
		- As efficent as you can get, just the elements themselves, (plus the array variable - a pointer)
